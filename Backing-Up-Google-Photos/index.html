Google Photos is wonderful. It backs up my photos and videos, tags them, and makes them available to share with my family.

But I also don‚Äôt trust that Google will never accidentally ‚Äúloose‚Äù my photos. Here is the process I use to back up my Google Photos data to a self-hosted instance of Photoprism.

This guide covers the process I use to back up my Google Photos content, in its original quality and with sidecar metadata, to Photoprism.

Prerequisites and Setup

My Photoprism instance runs on a TrueNAS Scale server as a kubernetes app. While yours doesn‚Äôt have to run in the exact setup, it‚Äôs worth describing. Photoprism has three primary storage folders, ‚ÄúOriginals‚Äù, ‚ÄúImports‚Äù, and a data path for sidecar files, multimedia caches, and the database file (if you‚Äôre using the built-in Sqlite storage, though I‚Äôd strongly caution you to set up an external MariaDB instance if you have a large photo library). Photoprism treats ‚ÄúOriginals‚Äù as essentially a read-only store of media - anything you put in here will be imported by Photoprism as-is. Meanwhile any media put in ‚ÄúImports‚Äù will be ingested by Photoprism and moved to the Originals folder; however you will have no control over the organization of these imported media. For our purposes, I strongly recommend putting your Google Photos media in ‚ÄúOriginals‚Äù to preserve the folder structure Takeout outputs.

In my TrueNAS setup the Originals and Import folders are mounted onto ZFS datasets on the host, which allows me to access and administer the media files while Photoprism isn‚Äôt running. Specifically, ‚ÄúOriginals‚Äù is mounted to /mnt/Bulk Storage/Photoprism/Originals, which we‚Äôll just call $ORIGINALS from here on. First and most importantly, we will be using ZFS extended attribute immutable to ensure that any files already in our ‚ÄúOriginals‚Äù folder won‚Äôt be overwritten when we extract the Takeout GZip archives. To do this, we can run the following command:

# Find all regular files in our Originals folder and mark them as +i (immutable).
find "$ORIGINALS" -type f -print0 \
| sudo xargs -0 -n 1 chattr +i

We‚Äôll also be using a few tools not distributed with TrueNAS Scale (or more Linux distros, for that matter). Most can be installed via package managers, the rest can be built from source:

  • pigz - Parallel GZip implementation (homepage).
  • progress - Coreutils‚Äô progress viewer (not necessary - just nice; homepage).
  • jdupes - Jody Bruchon‚Äôs enhanced fdupes clone (homepage).

Step 1 - Create Takeout Archives

  1. Go to Google Takeout.
  2. Click .
  3. Scroll down to ‚ÄúGoogle Photos‚Äù and select its checkbox.
  4. Scroll to the bottom and click .
  5. For ‚ÄúFile type‚Äù select .tgz, and for ‚ÄúFile size‚Äù select 50GB.
  6. Click .

Step 2 - Downloading Takeout Archives

Once your Google Takeout files are ready you will get an email (you can also check the Takeout site for completed archives). Depending on the size of our Google Photos library you may have multiple ~50GB files. I usually have around 10-15. I prefer to download one file at a time (depending on free disk space). Clicking the file will start a browser download, but I like to pause this download, right-click it and ‚Äúcopy download URL‚Äù, and then enter that into a download tool like Aria2.

The total size of your Takeout files may be much larger than your total Google Photos library. Takeout will organize your photos and videos by year and month, but any named albums you‚Äôve created will also have a top-level folder containing yet another copy of any photos or videos in that album.

Because of this, we‚Äôll be utilizing some ZFS tricks to minimize and deduplicate your disk usage.

Step 3 - Extracting Takeout Archives

Once I have some Takeout archive files downloaded (you don‚Äôt have to download them all at once, or even in order!) we‚Äôll start extracting them into our ‚ÄúOriginals‚Äù folder. Your Photoprism instance shouldn‚Äôt index Originals unless you tell it to, but I usually shut Photoprism down while performing these steps just to be safe.

Here's the script I use.

This will use pigz to extract the specified archive in parallel (for speed) and then extract the Google Photos files into our ‚ÄúOriginals‚Äù folder:

#!/usr/bin/env sh

# Usage:
# sudo ./extract-to-originals.sh takeout-archive.tgz

if [ "$(id --user)" -ne 0 ]; then
    echo "Please run with sudo!" 1>&2;
    exit 1;
fi

pigz --decompress <"$1" \
| tar --extract --strip-components=2 --skip-old-files --directory="$ORIGINALS" \
& progress --wait -monitor --command pigz;

chown -R photoprism:photoprism "$ORIGINALS";

Let‚Äôs break down what‚Äôs happening here:

  1. if [ "$(id -u)" -ne 0]; - checks if the script was run as root (or with sudo).
  2. pigz --decompress <"$1" - decompress the GZip file specified in the first passed to the script.
  3. tar --extract‚Ä¶ - extract files from the Tar archive that pigz decompressed. There‚Äôs a few extra options:
    • ‚Ä¶--strip-components=2‚Ä¶: remove some unnecessary path components that Takeout adds.
    • ‚Ä¶--skip-old-files‚Ä¶: skip files that already exist on disk.
    • ‚Ä¶--directory: extract files into our $ORIGINALS folder.
  4. progress --wait --monitor --command pigz - monitor the progress of the pigz command in decompressing the input file.
  5. chown -R photoprism:photoprism "$ORIGINALS" - Because tar attempts to preserve the ownership of the files it extracts, we need to correct this after extraction. Obviously update the user:group to suit your own needs.

Because the files already in our ‚ÄúOriginals‚Äù folder are marked immutable tar can‚Äôt overwrite them - and in fact we also provide tar with the --skip-old-files option to avoid a lot of error output about not being able to extract existing files. We do this so that the next time you want to back up your Google Photos, any files that existed between the last backup and the new one aren‚Äôt re-written and we can safely skip extracting them from the archive. This will save a ton of time.

Step 4 - Deduplicating Media Files

Repeat step 3 for all of the Takeout archives - you can delete each one as you go once it‚Äôs been extracted. Once you‚Äôre done, we can inspect how many files we have that are new by counting the files that are not immutable.

Here's a little script to do just that. 
#!/usr/bin/env sh

if [ "$(id -u)" -ne 0 ]; then
    echo "Please run with sudo!" 1>&2;
    exit 1;
fi

find "$ORIGINALS" -type f -print0 \
| xargs --null lsattr \
| grep -- "----------------------" \
| wc --lines;

Breaking it down:

  1. if [ "$(id -u)" -ne 0 ] - again, checks that we‚Äôre running as root or with sudo.
  2. find "$ORIGINALS" -type f -print0 - finds files in our originals folder. We use -print0 to safely handle filenames with weird characters in them.
  3. xargs --null lsattr - for each file that find outputs we want to run lsattr - this will list any extended attributes on each file.
  4. grep -- "----------------------" - for each line (file) of output from lsattr, we‚Äôre only interested in lines (files) that have no extended attributes set (files that have immutable set would show as ----i-----------------).
  5. wc --lines - count the number of lines (files).

Neat! Now let‚Äôs deduplicate all of these files. As mentioned in the aside above, Takeout will include multiple copies of each piece of media, as each one can appear in multiple albums. To deduplicate this and save you a bunch of hard drive space we‚Äôll use jdupes. This tool will search for files that are identical and (in our case) turn them into filesystem hardlinks - which means that the same data on disk can be pointed to by multiple file names.

First we need to change all files back from immutable. This can be done quickly with sudo chattr -R -i "$ORIGINALS". Then we run jdupes, which has a lot of options, but the ones we want are:

sudo jdupes --link-hard --recurse "$ORIGINALS"

Aka:

  • --link-hard - we want to hard-link all identical files together.
  • --recurse - consider all files within subdirectories of $ORIGINALS recursively.

When this is done we‚Äôll have potentially saved a lot of disk space!

Want to find out how much?

Here‚Äôs a short command that will tell you exactly how much space has been saved!

find "$ORIGINALS" -type f -links +1 -printf '%i %s\n' \
| awk 'a[$1]++{sum+=$2}END{print sum}' \
| numfmt --to=iec-i --suffix=B

Here‚Äôs the gist of what‚Äôs happening:

  1. find "$ORIGINALS" -type f‚Ä¶ - find all files in $ORIGINALS;
    • ‚Ä¶-links +1‚Ä¶ - that have more than one hardlink;
    • ‚Ä¶-printf '%i %s\n' - and output the file‚Äôs inode number and file size (in bytes).
  2. awk 'a[$1]++{sum+=$2}END{print sum}' - this line is a bit heavy, but here‚Äôs what the awk script is doing:

    • a[$1]++

      Increment the value stored in a[$1] (where $1 is going to be the inode number from each line). The inode is the unique data on disk - multiple file paths hardlinked to the same data will have the same inode number.

    • {sum+=$2}

      The {sum+=$2} part is a conditional clause - that means that if the value before it is ‚Äútruthy‚Äù it will execute. The first time a unique inode is encountered a[$1]++ will post-increment to 1, returning 0. This is falsy, so the conditional clause won‚Äôt run. Each time an inode is encountered after that, however, the value with be 1 or more, which is a truthy value and the conditional clause will run. All the clause actually does (when run) is increment the value sum (which starts at 0) by $2 (the file size in bytes).

    • END{print sum} - when the script ends, print the value in sum.

  3. numfmt --to=iec-i --suffix=B - this converts the value of sum output by awk to an IEC size (e.g., 1024 would be converted to 1.0KiB).

So as an example, say our ‚ÄúOriginals‚Äù folder only contains three files:

  • üñºÔ∏è a.jpg
  • üñºÔ∏è b.jpeg
  • üìÑ sidecar.json

And a.jpg and b.jpeg are both hardlinks to the same 1MiB of data, and sidecar.json is a 1KiB standalone file. When we run our script, the find portion will output three lines:

12345 1048576
12345 1048576
67890 1024

When awk gets these lines, it will perform the following:

  • Line 1: Post-increment a[12345] (from 0 to 1), returning 0. 0 is falsy, so {sum+=1048576} doesn‚Äôt run.
  • Line 2: Post-increment a[12345] (from 1 to 2), returning 1. 1 is truthy, so {sum+=1048576} runs - sum is now 1048576.
  • Line 3: Post-increment a[67890] (from 0 to 1), returning 0. 0 is falsy, so {sum+=1024} doesn‚Äôt run.
  • End of input: END{print sum} runs, outputting 1048576.

Finally, numfmt converts 1048576 to 1.0MiB, and we see our space savings is 1 megabyte! This is correct, as if b.jpeg wasn‚Äôt hardlinked to a.jpg it would take an additional 1 megabyte.

Cleanup

Finally, we can re-mark our files as immutable:

find "$ORIGINALS" -type f -print0 \
| sudo xargs -0 -n 1 chattr +i

Now we turn Photoprism back on, and tell it to re-index new files in the Originals folder. It will even intelligently use the sidecar JSON files that Takeout exports to enrich each media item with additional metadata.
