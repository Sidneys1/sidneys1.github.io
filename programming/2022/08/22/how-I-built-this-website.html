Getting Started With Jekyll

I use a Windows PC, and I find that the Jekyll environment is easier to set up under the Windows Subsystem for Linux, so that‚Äôs what I‚Äôll be using going forward in this post. First, we‚Äôll install Ruby via your package manager - Ubuntu 20.04 has Ruby 2.7 available on apt via apt install ruby. Next we‚Äôll use the Ruby package manager, gem to install both Jekyll and Bundler: gem install bundler jekyll. Finally, you can create a new Jekyll site with jekyll new sitename.

Once inside the new site folder, you can generate and serve your page with bundle exec jekyll serve! This will create a development server on http://localhost:4000. Let‚Äôs quickly talk about how Jekyll works:

  1. You create Markdown or HTML files in your site‚Äôs folder for each post you wish to make.
  2. Jekyll processes these files (along with some templates - we‚Äôll get to these later) to generate static HTML files.
  3. These static HTML files (in _site) are what you want to put on your web hosting! It‚Äôs all static, so you don‚Äôt need anything fancy like server-side PHP or client-side Angular or React.

To simplify the build-preview and build-release process, let‚Äôs make a simple Makefile. GNU Make is a tool that allows you to define simple build steps, and then figures out the interdependencies between these steps for you. Let‚Äôs take a look:

	bundle exec jekyll serve --watch --livereload --force_polling --drafts --destination _site_live/

	env JEKYLL_ENV=production bundle exec jekyll build

clean:
	rm -rf _site/

Let‚Äôs break down each of these sections:

  • all: build: ‚Äúall‚Äù is the default rule run when you execute make. This line says when ‚Äúall‚Äù is run, run ‚Äúbuild‚Äù first.
  • build: _site/: when the ‚Äúbuild‚Äù rule is run, we depend on ‚Äú_site/‚Äù being built first.
  • serve: ...: If you execute make serve, this rule is triggered. Make will run the bundle exec jekyll serve ... command we‚Äôve listed. The parameters included are:
    • --watch: live-rebuild any changes made to the posts as you edit them (very useful!).
    • --livereload: use LiveReload to automatically refresh your browser when changes are rebuilt.
    • --force_polling: this is a workaround for some WSL bugs. See this WSL issue.
    • --drafts: include posts under the _drafts folder (this is where you can put in-progress posts).
    • --destination _site_live/: this directs the serve command to build the static output under a separate folder than the normal _site/ rule. This way we can‚Äôt accidentally publish our live preview version of the website - if we did, it would be broken because all links would lead to localhost instead of your website‚Äôs URL!

  • _site/: ...: this rule matches a file path, _site, which is Jekyll‚Äôs output folder. Make understands file paths and will know that this rule will build the contents of _site. Note that we use env to tell Jekyll to use the ‚Äúproduction‚Äù environment.

    We‚Äôll never run this rule manually (though you could with make _site/), but our build rule depends on this running, and so Make will automatically run it when needed.

  • clean: ...: Run make clean to invoke this rule - in our case, we just delete the _site/ folder.

Modifying the Jekyll Configuration

Jekyll configuration is stored in the _config.yml file at the root of our site. There are a couple things in here we‚Äôll want to adjust before we publish our site to the world wide web. First of course we‚Äôll want to set a few basics:

  • title: The name of your site (shown in the header).
  • email: A contact email address (shown in the footer).
  • url: This one is important - this sets the url used when clicking on absolute links to other pages within your site!
  • baseurl: Used in conjunction with url. The format used is {url}{baseurl}/path/to/page.html. If you‚Äôre hosting nothing but Jekyll on your website, then leave this blank.

And that‚Äôs about it! We‚Äôll dig into configuration more when we talk about hosting on Tor and IPFS.

Hosting

Ok! Now that we have a (basic) site, how do we host it? Let‚Äôs start simple and look at NearlyFreeSpeech.net, whom I‚Äôve been using for years. They‚Äôre cheap, no-nonsense, and don‚Äôt have any crazy hidden fees. They also have support for Let‚Äôs Encrypt certificates, making SSL support both easy and automatic. Head on over and set up an account. It‚Äôll walk you through creating a site (your website hosting) and you can even register and connect the domain name all within their system. Eventually you‚Äôll find the settings to be able to connect with SSH - enable this and let‚Äôs update our Makefile again. Add this rule to your Makefile:

	rsync --itemize-changes --checksum --recursive --compress --delete _site/* $USER@ssh.phx.nearlyfreespeech.net:.

Let‚Äôs break it down again:

  • publish: _site/: when we execute make publish, we want to make sure the rule to build the _site/ folder runs first.

  • rsync [...] _site/* $USER@ssh.phx.nearlyfreespeech.net:.: we‚Äôll use rsync (a remote file-copying tool) to connect to our hosting via SSH, synchronizing the contents of _site with the root (.) folder of our hosting. You‚Äôll want to replace $USER with your NearlyFreeSpeech.net username, of course.

    The options we‚Äôre using are:

    • --itemize-changes: list changes as we go (for clarity).
    • --checksum: rely on a file checksum (not timestamps and file size) to determine if files need to be replaced.
    • --recursive: recurse into subdirectories of _site/.
    • --compress: HTML is very compressible, so why not save some bandwidth?
    • --delete: this allows rsync to remove files on your hosting that are no longer present in your _site folder. This can be useful if you rename or delete a file that you no longer want people to be able to access.

Running make publish now should build your website and then prompt your for your NearlyFreeSpeech.net password to connect with SSH. Provide your password and watch as your site is made available!
